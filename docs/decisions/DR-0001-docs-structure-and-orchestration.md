# DR-0001 — 문서 체계 분리 + 멀티모델 오케스트레이션

- 날짜: 2026-07-04
- 상태: 채택
- 근거: 사용자 지시 (의사결정·변경로그·프로젝트 문서 용도별 분리 / Fable=계획·사고, opus·sonnet=실행)

## 맥락
SKAI(D4D 해커톤 T2 · Air ISR Fusion Copilot)는 24H 해커톤 프로젝트로, 루트에 기획 문서 8개가 SSOT로 존재. 실행이 시작되면 (a) 왜 그렇게 결정했는지, (b) 아키텍처·온톨로지가 어떻게 변해왔는지, (c) 각 단계에서 실제로 무엇을 했는지가 섞이면 추적 불가능해짐. 심사 기준(Technical Execution·Deployability)에도 변경 추적성이 유리.

## 결정
1. **3계층 문서 분리**:
   - `docs/decisions/DR-NNNN-*.md` — 의사결정 기록 (맥락→결정→기각 대안→영향)
   - `docs/CHANGELOG.md` — 아키텍처·온톨로지 변경로그 (온톨로지 v0.1 기준선부터)
   - `docs/worklog/` — P0~P6 단계별 실행 로그 (에이전트 상세 리포트 + 종합)
   - 루트 8문서는 기획 SSOT로 유지, 변경 시 CHANGELOG 기록 의무.
2. **오케스트레이션**: Fable이 계획·검토·문서화, 실행(코드·API 호출·빌드)은 opus/sonnet 서브에이전트에 위임. 개인 규칙은 `CLAUDE.local.md`(gitignore)에 — 팀 저장소 오염 방지.

## 기각 대안
- 단일 WORKLOG.md에 전부 기록 — 용도 혼합으로 검색성·심사 제시성 저하. 기각.
- 팀 공유 CLAUDE.md에 오케스트레이션 규칙 추가 — 사용자 개인 도구 설정(모델 분담·codex 폴백)이 팀 문서에 노출됨. 기각, CLAUDE.local.md로.

## 영향
- 이후 모든 DR은 이 번호 체계를 따름. 온톨로지 변경은 CHANGELOG + 스멜테스트 재통과 확인이 세트.
